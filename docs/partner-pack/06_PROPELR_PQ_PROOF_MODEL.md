# Propelr PQ Proof Model

## Purpose

The PQ proof layer provides a durable, queryable, API-native proof of payment execution behavior without using blockchain.

## Proof Generation Flow

1. payment request received
2. payment attempt executed
3. proof request assembled
4. PQ sidecar returns proof id
5. proof metadata persisted in Proof Vault
6. proof status exposed by merchant reference
7. proof retrieved and verified by proof id
8. cryptographic receipt generated by API

## Current Stored Fields

Proof Vault currently stores:
- `proof_id`
- `payment_attempt_id`
- `merchant_id`
- `merchant_reference`
- `proof_hash`
- `hash_algorithm`
- `signature`
- `signature_algorithm`
- `proof_status`
- `request_fingerprint`
- `processor_response_fingerprint`
- `sidecar_version`
- `evidence_bundle_uri`
- `policy_snapshot`
- `created_at`
- `verified_at`

## Current Proof Modes

### Async Audit
- payment is not blocked
- proof is submitted in the background

### Strict
- payment requires successful proof submission
- failure to submit proof blocks execution

## Receipt Model

Receipt payload includes:
- proof id
- payment attempt id
- merchant reference
- proof hash
- policy snapshot
- sidecar version
- request fingerprint
- verification-relevant metadata

## Current Note

The current sidecar implementation is a controlled stub and is intended to be replaced with a production proof engine in the next integration phase.
